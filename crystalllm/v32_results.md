# CrystaLLM v32 — 同步扩规模 (失败)

> **Q: 同步扩大规模 (drafter 28M→100M, verifier 28L→32L), 能进一步加速吗?**
> **A: 不能. v32 速度 385ms (vs v31 206ms, **慢 87%**). 接受率 92.5% (vs 95.5%, 略降).**

## TL;DR

| 指标 | v31 | **v32** | 变化 |
|---|---:|---:|---:|
| **SpS 速度** | 206ms | **385ms** | **+87%** |
| **AR 速度** | 564ms | 904ms | +60% |
| **接受率** | 95.5% | **92.5%** | -3% |
| **PPL** | 2.39 | 2.37 | -0.7% |
| **Verifier** | 28L (555M) | 32L (633M) | +14% |
| **Drafter** | 28M | 100M | 4.2x |

**核心发现**:
- ❌ **同步扩规模失败**: v32 比 v31 慢 87%
- ❌ **更大 drafter 没帮助**: 接受率反而略降 (过拟合)
- ❌ **更大 verifier 没显著改善 PPL**: 仅 -0.7%
- ✅ **v31 (28M + 28L) 仍是当前架构最优**

## 1. 设计思路

### 1.1 v31 成功后的下一步

v31 证明"扩散 drafter + AR verifier"框架有效:
- 速度 206ms (2.73x 加速)
- 接受率 95.5%
- 接受 tokens/round ~7.6

**推测**: 更大 drafter 更接近 verifier, 接受率更高; 更大 verifier PPL 更低. 同步扩大可能进一步加速.

### 1.2 实际结果

| 假设 | 实际 |
|---|---|
| Drafter 100M 比 28M 接受率高 | **92.5% vs 95.5%, 反而降** |
| Verifier 32L 比 28L PPL 低 | **2.37 vs 2.39, 仅 -0.7%** |
| 总加速比 v31 更快 | **385 vs 206ms, 慢 87%** |

## 2. 失败原因

### 2.1 Drafter 100M 过拟合

**100M drafter pred_match 仅 56%** (vs v31 28M 73%):
- 更大模型过拟合了 8-token 短序列模式
- 28M 已经足够拟合 z → 8 tokens 映射
- 再大反而损害泛化能力

**证据**:
- v32 训练: pred_match 56% (最终)
- v31 训练: pred_match 73% (最终)
- 28M 模型泛化更好

### 2.2 Verifier 32L 收益递减

**v32 verifier PPL 2.37 vs v28.5 2.39**:
- 仅 -0.7% 改善
- 但速度 904ms vs 544ms (+66%)
- BAD-DP 架构限制: decoder 只看 z, 不看 prefix

**证据**:
- v28 (28L) 报告: "加深度收益递减 (-3%)"
- v32 (32L): "进一步递减 (-0.7%)"
- 边际收益已接近 0

### 2.3 同步扩大 = 双重代价

每个组件的代价叠加:
- Drafter 100M: +10ms/forward
- Verifier 32L: +3ms/forward
- **每 round 多 13ms, 但 rounds 没减少**

## 3. 速度分解对比

| 步骤 | v31 | v32 | 差异 |
|---|---:|---:|---:|
| Prior 采样 | 7.05ms | 9.17ms | +30% |
| 扩散 drafter (K=8) | 7.98ms | **18.51ms** | **+131%** |
| Verifier 1 forward | 6.97ms | **10.43ms** | **+50%** |
| **每 round** | **22ms** | **38ms** | **+73%** |
| Rounds | 20 | 14 | -30% |
| 总时间 (估) | 440ms | 532ms | +21% |
| 实测 | 206ms | 385ms | +87% |

**关键**: rounds 数减少 (20→14, -30%) 部分抵消了 per-round 增加, 但实测仍慢 87%.

## 5. 反思: 同步扩规模的失败

### 5.1 不是规模问题, 是架构问题

v31 → v32 揭示:
- 28M drafter + 28L verifier 已经是**当前架构下的最优点**
- 进一步扩大规模不解决问题
- 真正问题在 **BAD-DP 架构**: decoder 只看 z, 不看 prefix

### 5.2 "适度规模"原则

每个组件有不同的"最优规模":
- Drafter: 够拟合 z → K tokens 即可 (28M 已够)
- Verifier: BAD-DP 限制下 28L 与 32L 几乎一样
- 数据: 19K 已接近当前上限

**同步扩规模 = 把每个组件都推过最优, 反而变差**.

### 5.3 与 v31 对比的关键差异

| | v31 (优) | v32 (过扩) |
|---|---|---|
| Drafter | 28M (够) | 100M (过拟合) |
| Verifier | 28L (BAD-DP 限制下够) | 32L (无收益) |
| 总效果 | **2.73x 加速** | **1.41x 加速** |

**适度规模 > 过度规模**.

## 6. 下一步建议

### 6.1 短期 (v33): 回退到 v31 框架

**v33 = v31 优化版**:
- 28M drafter (与 v31 同)
- 28L verifier (v28.5)
- 优化方向:
  - 数据扩展 (v24 + 高质量子集)
  - drafter 训练步数 (4000 → 6000)
  - 增加 K (8 → 16, 试更大草稿)
  - 优化 Python overhead

### 6.2 中期 (v34): 架构突破

**v34 = BAD-DP → 标准 decoder**:
- decoder 看 z + prefix (像 GPT)
- 这是 BAD-DP 的根本修复
- 预期 PPL 2.39 → < 2.0

### 6.3 长期 (v35+): 模型规模与架构联合突破

- 7B+ 模型 (KV cache 重新有用)
- 标准 decoder (z + prefix)
- 扩散 + AR 深度融合

## 8. 总结

**v32 同步扩规模失败**:
1. ❌ Drafter 28M → 100M: 接受率反而降 (95.5% → 92.5%)
2. ❌ Verifier 28L → 32L: PPL 仅改善 -0.7%
3. ❌ 总速度 206ms → 385ms (+87% 慢)
4. ✅ 验证了 v31 框架的**适度规模最优**

**关键洞察**: "同步扩大"不等于"更好". 在当前 BAD-DP 架构下, v31 的 28M + 28L 已经是平衡点.

**当前 SOTA**: **v31 SpS (206ms, PPL 2.39)**

**下一步**: 回退到 v31, 优化数据/训练步数, 或彻底改架构 (BAD-DP → 标准 decoder).
