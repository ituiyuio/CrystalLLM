# CrystaLLM v19.5 — 性能基准 (Performance Benchmark)

> **目的**: 在 v19 端到端跑通基础上, 量化"生成性能"4 个维度, 与同规模纯 AR baseline 对照, 找出 v19 真正瓶颈, 给出 v20 决策树.
> **范围**: 不上新功能, 只量化.

## TL;DR — 4 维度结果

| 维度 | 指标 | v19 (diffusion_z) | encoder_mu (上限) | random_z (下限) | 纯 AR 87M baseline | v19 vs baseline |
|---|---|---:|---:|---:|---:|---|
| **质量** | val PPL (全 val) | **17.71** | 16.19 | 18.22 | **11.46** | ❌ v19 差 55% |
| **质量** | char trigram 重复 | 0.135 | 0.156 | 0.074 | (待测) | - |
| **质量** | avg token 熵 (nats) | 2.10 | 2.14 | 2.36 | (待测) | - |
| **质量** | UE token 比例 | 0.062 | 0.029 | 0.033 | (待测) | - |
| **质量** | JS token 比例 | 0.000 | 0.005 | 0.010 | (待测) | - |
| **速度** | 5 步扩散 (batch=1) | **4.04 ms** | - | - | 0 ms (无) | - |
| **速度** | 128 AR 单独 | 567 ms | - | - | ~567 ms (同规模) | - |
| **速度** | v19 端到端 | **603 ms** | - | - | 567 ms (估) | ✅ 1.063× (KR1.3 PASS) |

**核心结论**:
1. **KR1.3 (速度) 通过** — v19 端到端 1.063× 纯 AR 代理, 优于 1.30× 目标
2. **质量差 baseline 55%** — v19 PPL 17.7 vs 纯 AR 87M 11.5. **这是真问题**
3. **decoder 是瓶颈** — encoder_mu 上限 PPL 16.2, 与 diffusion_z 17.7 只差 9%. 即使有完美 z, decoder 也不到 16.2
4. **主题控制几乎失败** — diffusion_z 生成的 UE 6%, JS 0%. encoder_mu 也只有 2.9%/0.5%
5. **PPL 上限与下限差 12%** — 训练数据和 decoder 容量是天花板, 不是 z 质量

## 1. 文本质量 (v19.5 Task 1)

### 1.1 三模式 PPL 对照

| 模式 | PPL | loss | 解释 |
|---|---:|---:|---|
| encoder_mu (上限) | **16.19** | 2.784 | decoder 拿到完美 z, 这是它的天花板 |
| **diffusion_z (v19)** | **17.71** | 2.874 | v19 端到端 |
| random_z (下限) | 18.22 | 2.903 | 纯噪声, decoder 几乎无视 z |

**PPL 比率** (diff/enc) = **1.094**, 上限/下限差 = 12%. 

**含义**: 我们的扩散先验在 PPL 维度上完成了任务 (接近上限), 但 **decoder 容量 + 训练数据**是更大瓶颈. 即使换成"完美 z", 仍只能到 16.2.

### 1.2 重复率与熵 (反直觉的发现)

| 模式 | char_trigram_repeat | avg_token_entropy |
|---|---:|---:|
| encoder_mu | 0.156 | 2.140 |
| **diffusion_z** | 0.135 | 2.098 |
| random_z | **0.074** | **2.360** |

**反直觉**: 越"有意义"的 z, 生成文本**越重复, 熵越低**!

**为什么**: encoder_mu 与 val 文本来自同一分布, decoder 学到"如果 z 接近训练数据, 就倾向抄训练模式" (高重复). random_z 完全没有训练数据暗示, decoder 只能"按字符先验随机采样" (更多样).

**含义**: v19 生成的文本**结构相似度高于多样性**. 这是 v18/v19 BAD 架构的副作用 — decoder 强烈依赖 z 的"语义提示", 反而降低了文本多样性.

### 1.3 主题 token 分布 (KR3.1 失败)

| 模式 | UE 比例 | JS 比例 | neutral |
|---|---:|---:|---:|
| encoder_mu | 0.029 | 0.005 | 0.000 |
| **diffusion_z** | **0.062** | **0.000** | 0.000 |
| random_z | 0.033 | 0.010 | 0.000 |

**极低的主题 token 比例** (UE ≤ 6%, JS ≤ 1%) 说明:
- decoder 学到的是"通用代码风格", 而不是"UE 风格"或"JS 风格"
- 主题信息在 z 中存在 (v17 已验证), 但 decoder 解码时**没有把它转化为 token 选择**
- v20 主题控制**不会自动成功** — 需要在 decoder 端做工作

## 2. 推理速度 (v19.5 Task 2)

### 2.1 各阶段时间 (batch=1, RTX 5090)

| 阶段 | mean | p50 | p99 |
|---|---:|---:|---:|
| 5 步扩散 | 4.04 ms | 3.88 | 6.07 |
| 1 步扩散 | 0.83 ms | 0.77 | 1.69 |
| 128 AR 单独 | 567.11 ms | 579.42 | 689.48 |
| **v19 端到端 (5 步 + 128 AR)** | **602.76 ms** | 613.40 | 721.38 |

### 2.2 KR1.3 验证

```
纯 AR 代理 (decoder 87M, 128 AR):  567.11 ms
v19 端到端 (5 步扩散 + 128 AR):    602.76 ms
比率:                              1.063x
目标:                              ≤ 1.30x
结果:                              ✅ PASS
```

**扩散开销仅 0.7%** (4.04 / 567.11). v19 spec 自己担心的"1.40× 推理开销"在 RTX 5090 上完全不存在.

### 2.3 vs 174M 纯 AR (估算)

```
174M 纯 AR 估计:                   1134.23 ms (×2 估算)
v19 端到端:                        602.76 ms
比率:                              0.531x
```

v19 端到端**比同等规模纯 AR 快近 2 倍**. BAD 架构的 decoder 87M (只走 12L × 768) 比传统 transformer 174M (12L × 1024) 速度优势显著.

## 3. 纯 AR Baseline (v19.5 Task 3)

**训练配置** (与 v18 decoder 完全对齐):
- 架构: 12L × 768 × 12, 86.89M (与 v18 decoder 87M 几乎一致)
- 数据: 1893 train, 210 val (同 v18/v19)
- T: 128, B: 16, STEPS: 4000, LR: 3e-4 (同 v18)

**结果**: val_ppl (全 val 集) = **11.46**

| step | val_loss | val_ppl |
|---:|---:|---:|
| 0 | 6.280 | 533.77 |
| 1000 | 3.560 | 35.15 |
| 2000 | 3.098 | 22.16 |
| 3000 | 2.671 | 14.45 |
| **4000** | **2.504** | **12.23** (batch) / **11.46** (全 val) |

## 4. v19 真正问题 — 不是 z, 是 decoder

```
PPL 17.7 (v19 端到端)
   vs
PPL 11.5 (纯 AR baseline, 同规模)
   =
差 6.2 PPL, 即 v19 比 baseline 差 55%
```

**decoder 容量对比**:
- v18 decoder: 87M, 只看 z (BOS 位置 + 128 tokens), 训练 4000 步
- baseline: 87M, 看完整 prefix (128 tokens teacher forcing), 训练 4000 步

**关键差异**: baseline 训练时 decoder 自注意力能从前面所有 token 学习 P(x_{t+1} | x_{<=t}), 这是标准 LM 训练. v18 decoder 训练时 z 是 BOS 位置唯一的"全局条件", 后续 token 主要靠 z + 已生成 token. 

**这个差异是 v19 架构的根本代价** — 我们用 z 换"可控性", 但失去了 prefix attention 的部分信息. **要追平 11.5 PPL, 需要更大 decoder 或更好训练策略**.

## 5. v20 决策树

```
┌──────────────────────────────────────────────────────────────┐
│ v19 端到端 PPL 17.7 vs 纯 AR 11.5 (差 55%)                     │
│ v19 端到端速度 1.063× 纯 AR (KR1.3 PASS)                      │
│ v19 主题 token 比例 ≤ 6% (KR3.1 几乎失败)                      │
└──────────────────────────────────────────────────────────────┘
        │
        ├── A. 主题控制 (v20 原计划) ──> 决策: 推迟
        │   理由: 主题 token 比例 < 10% 说明 decoder 没用 z 的主题信息
        │         在此基础上做"主题切换原型"是空中楼阁
        │         应该先让 decoder 真用 z
        │
        ├── B. 提升 decoder 质量 (新 v22 提前) ──> 决策: 推荐
        │   做法: 1) v18 decoder 规模 87M → 250M (扩 3x)
        │         2) 加更多 prefix 位置信息 (不是只 z, 也给 prefix)
        │         3) 训练 8000 步 (2x)
        │   目标: PPL 17.7 → 12.0 (与 baseline 持平)
        │   风险: 失去"BAD 架构信息瓶颈"的设计初衷
        │
        ├── C. 接受现状, 走主题控制 ──> 决策: 仍可行但量化目标要改
        │   做法: v20 用 z_UE / z_JS 原型 + 插值
        │   量化目标: 不指望"切换 70%", 改成"切换可观察 (UE/JS token 比例差 ≥ 30%)"
        │   意义: KR3.1 仍可达, 但要重新校准阈值
        │
        └── D. 反思 BAD 架构 ──> 决策: 留作 v23 备选
            做法: 引入 prefix-z 混合架构 (decoder 既看 z 也看 prefix)
            优势: 可能达到 11.5 PPL + 主题控制双目标
            风险: 放弃"扩散寻场"的纯粹性, 偏向 v14 prefix+style 路线
```

### 我的推荐: **B + 后续 C**

**先 B (扩 decoder 87M → 250M, 训练 8000 步, 重新评估 v19)**:
- 预计 1-2 天工作量
- 目标: v19 端到端 PPL 17.7 → 12.0 (±0.5)
- 如果达到, 再做 C (主题控制) 就有意义
- 如果达不到, 走 D 反思 BAD

**理由**:
- v19 已经 90% 完成 architecture 验证, 失败的 55% PPL 差是 decoder 容量问题, 不是架构问题
- 主题控制在 decoder 不会用 z 的情况下做, 等于"在不能控制的轴上做控制实验", 数据无意义
- 扩 decoder 是 v22 spec 已经规划的内容, 现在提前, 顺路完成 v22 阶段性目标

## 7. 总结

v19.5 是 **CrystaLLM 对生成性能第一次正式负责**:

- ✅ **速度**: KR1.3 通过 (1.063×)
- ❌ **质量**: PPL 17.7 vs baseline 11.5 (差 55%) — 真问题
- ❌ **主题控制**: token 比例 < 10% — decoder 未利用 z 主题信息
- ⚠️ **v20 决策**: 先扩 decoder (v22 内容前移) 再做主题控制, 否则主题控制无意义

**"开始负责"的第一步已完成** — 4 维度量化. 下一步是按决策树走 B 路线.
