# v28 全面 Scale Up 设计

> **Q: 把 decoder (500M) + encoder + 数据 (19K) 全部 scale up, 能把 PPL 从 2.44 推到多少?**
> **A: 目标 PPL < 1.8, 模型 1B+, 数据 100K+. 全面 scale up.**

## 1. 背景

### 1.1 v23-v27 现状

| 版本 | 模型规模 | 数据量 | PPL | 备注 |
|---|---:|---:|---:|---|
| v23 | 500M | 6K | 8.86 | 早期 |
| v24 | 500M | 19K | 3.28 | 数据增强 |
| v25 | 500M T=512 | 19K | 2.44 | T 扩展 |
| v26 | 100M drafter | 19K | 4.55 | 投机解码 |
| v27 | (500M + KV gen) | 200 | (失败) | 分布偏移 |

### 1.2 当前瓶颈

1. **数据量**: 19K 太少 (avg len 3835 字符 = ~1000 tokens)
2. **模型规模**: 500M 在 ~3-4B tokens 上可能欠拟合
3. **PPL 2.44 仍然有改进空间**: GPT-2 small (117M) 在大数据上可达 < 1.5 PPL

### 1.3 v28 动机

**全面 scale up**:
- 数据: 19K → 100K+ (extended_v23 有 1.1M)
- 模型: 500M → 1B+
- 训练: 更长训练, 更精细的 LR schedule
- 目标: PPL 2.44 → < 1.8

## 2. 架构设计

### 2.1 模型规模对比

| 配置 | 参数量 | Layers | embd | head | head_dim |
|---|---:|---:|---:|---:|---:|
| v25 (当前) | 476M | 24 | 1280 | 20 | 64 |
| **v28 目标** | **~1.2B** | **32** | **1600** | **25** | **64** |
| v28 大版本 | ~1.5B | 32 | 1792 | 28 | 64 |

**选择**: v28 = 32L × 1600 × 25 (~1.2B)
- 参数量 2.5x v25
- 训练时间预估: 1.5-2h (16 GPU min)
- 推理速度: 估计 1.5-2x v25 (~750-1000ms)

### 2.2 数据策略

| 数据源 | 样本数 | 平均长度 | 用法 |
|---|---:|---:|---|
| v24 train (现有) | 19,307 | 3835 chars | 复用 |
| **extended_v23** | **1,131,427** | 468 chars | **新增主数据** |
| v23 train | 6,317 | 4294 chars | 补充 |

**关键问题**: extended_v23 是代码 + 短文本 (468 chars), 与 v24 (Python 库 3835 chars) 不同.

**决策**:
- 用 extended_v23 **前 50K 样本** 作为新数据
- 与 v24 train 合并: **19K + 50K = 69K 样本**
- 总字符数: 19K×3835 + 50K×468 = 73M + 23M = **96M chars** (~26M tokens)

### 2.3 训练配置

| 参数 | 值 | 备注 |
|---|---|---|
| B | 8 | 16 GPU 显存不够 |
| T | 512 | 沿用 v25 |
| LR | 5e-5 (1B) | 1B 模型用小 LR |
| STEPS | 8000 | 16 GPU min @ 8K |
| Warmup | 800 | 10% |
| LR sched | cosine | |
| 数据增强 | 切窗 ×2 | |

### 2.4 增量 vs 全量训练

- v25 24L×1280×20 → v28 32L×1600×25
- 维度不同, **不能 warm-start**
- 从零训练, 但用 v25 的训练超参作为参考

## 3. 实施步骤

### 3.1 数据准备 (~30 min)

```python
# build_v28_data.py
- 加载 extended_v23.parquet (1.1M 短文)
- 过滤长度 200-1500 chars
- 取前 50K 样本
- 与 v24_train.parquet 合并
- 输出 v28_train.parquet (~70K)
- 同样方式生成 val
```

### 3.2 Encoder 训练 (~30 min)

是否训练新 encoder?
- v24 encoder 是 100M, 256 维 z
- 数据从 19K → 70K, encoder 可能欠拟合
- **决策**: 复用 v24 encoder, 不重新训练 (节省时间, encoder 主要影响 z 质量)
- 如果 PPL 不达标, 再考虑

### 3.3 Decoder 训练 (~1.5-2h)

```python
# train_v28_decoder.py
- 32L × 1600 × 25 (~1.2B)
- 8000 步, B=8, T=512, LR=5e-5
- 16 GPU min (warmup) + ~2h 训练
- 复用 v24 diffusion prior + cached z
```

### 3.4 评估 (~15 min)

- PPL on v24 val
- 速度 (AR 100 tokens)
- 与 v25 (2.44, 502ms) 对比

## 4. 风险

### 4.1 GPU 显存

- 1.2B 模型, fp32: 4.8GB params
- AdamW (2x): 9.6GB
- 激活 (B=8, T=512, 32L): ~10GB
- **总计: ~20GB**, 在 34GB 显存下 OK

### 4.2 训练时间

- 8000 步 × B=8 × T=512 = 32M tokens
- RTX 5090 ~1 TFLOPS fp16: 32M × 1.2B × 6 (forward+backward) = 230 TFLOP
- 230 TFLOP / 1000 TFLOPS = 0.23s × 8000 步 = **30 min** (理论)
- 实际 ~1.5-2h (考虑 launch overhead)

### 4.3 PPL 不达标

可能原因: 数据质量 (extended_v23 短文本), 模型容量不够, 训练步数不够.

**对策**:
- 先训练, 看 PPL
- 如果 PPL > 2.0, 增加 STEPS 或模型大小

## 5. 评估标准

| 指标 | 目标 | 最低 | 备注 |
|---|---|---|---|
| PPL | < 1.8 | < 2.0 | 主要指标 |
| 速度 | < 1500ms | < 2000ms | 与 v25 比 1.5-2x 慢可接受 |
