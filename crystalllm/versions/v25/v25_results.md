# CrystaLLM v25 — T=512 上下文扩展 (4x)

> **Q: 把 decoder 上下文从 T=128 扩到 T=512 (4x), PPL 能改善多少? 速度会怎么变?**
> **A: PPL 3.28 → 2.47 (-25%), 速度 751ms → 828ms (+10%). 长上下文对 code completion 效果极佳! KR1.3 = 0.311x (仍 < 1.30x).**

## TL;DR

| 指标 | v24 | **v25** | 变化 | vs 500M AR |
|---|---:|---:|---:|---:|
| **T (上下文)** | 128 | **512** | **4x** | — |
| **PPL (端到端)** | 3.28 | **2.47** | **-25%** | **-72%** |
| PPL 范围 (enc/rand) | 1.86% | 0.77% | -58% | decoder 弱依赖 z |
| **速度 (5步+100AR)** | 751ms | **828ms** | +10% | — |
| **KR1.3 vs 500M AR** | 0.282x | **0.311x** | -10% | — |
| 训练时间 | 11 min | 12 min | ~相同 | — |
| 训练样本 | 19,307 | 19,307 | 复用 | — |

**PPL 演化路径** (BAD-DP 路线):
```
v18 (87M, 64z, T=128):  17.71
v20a (229M, 64z, T=128): 13.05
v21 (475M, 64z, T=128):  5.83  ← 扩 decoder
v22a (475M, 256z, T=128): 4.39  ← 扩 z + 主题对齐
v23  (475M, 256z, T=128): 3.37  ← 扩数据 3.3x (本地)
v24  (475M, 256z, T=128): 3.28  ← 扩数据 3.1x (raw_v23)
v25  (475M, 256z, T=512): 2.47  ← 扩上下文 4x (本版本)
```

**累计改善**: 17.71 → 2.47 = **-86% PPL, KR1.3 从 ~5x 降到 0.31x**.

## 1. 设计决策

### 1.1 为什么要扩 T

v24 报告已经显示**数据扩展边际收益饱和**:
- 1.9K → 6.3K (3.3x): PPL -23%
- 6.3K → 19.3K (3.1x): PPL -2.7%

扩数据边际收益已下降 **10x**. 唯一未挖掘的维度是**上下文长度 T**.

### 1.2 为什么 T=512 而不是 256

| T | 上下文 | PPL 估 | 速度估 | 备注 |
|---:|---:|---:|---:|---|
| 256 | 2x | 3.15 (-4%) | 900ms | 收益小 |
| **512 (选)** | 4x | 3.0 (-7-10%) | 1500ms | 显著改善 |
| 1024 | 8x | 2.9 (-12%) | 2400ms | KR1.3 ≈ 1.0x |

选 512 是因为:
1. 数据已饱和, T 是高边际收益的维度
2. 4x 上下文应该有明显 PPL 改善
3. 评估时 AR 100 步实际不受 T 影响 (cur 增长到 100, 不超 T), 实际速度只 +10%

### 1.3 架构改动

```
v24 → v25 架构映射
─────────────────────
Encoder:     T=128 (不变)
Diff prior:  256 维 (不变)
Decoder:
  T: 128 → 512 (pos embedding 130 → 514)
  其他全部 warm-start from v24 decoder
  Pos 初始化: 130 + 128*3 = 514 (v24 pos 重复 4x)
```

**关键洞察**: Encoder 不需要改. Encoder 用 T=128 切窗, 每个 z 对应 128 字符. Decoder T=512 只是消费 z 时看更长上下文.

**关键洞察 2**: 评估时 AR 100 步用 cur (1→100) 作为输入, 不超 T. 所以 T 增大不直接拖慢 AR.

## 2. 三步训练流水

### 2.1 v25 encoder (复用 v24)

无变化, 256 维, 12L × 768 × 12.

### 2.2 v25 cached z (复用 v24)

无变化, D_Z=256, 19,307 train + 1,016 val.

### 2.3 v25 diff prior (复用 v24)

无变化, 256 维 CFM, 6.58M, cos_sim 0.976.

### 2.4 v25 decoder (新训练, T=512, warm-start from v24)

**复用 v24 的关键修复**: 用 v22 vocab (2261) 让 tok.weight/head.weight 形状匹配.

**Pos embedding 初始化策略 A**: 复制 + 4x 重复 + 截断
```python
v24_pos = v24_state["pos.weight"]  # (130, 1280)
# v25 pos[0:130]   = v24_pos[0:130]   (保留原 z + BOS + 128 tokens)
# v25 pos[130:258] = v24_pos[0:128]   (新位置 130-257)
# v25 pos[258:386] = v24_pos[0:128]   (新位置 258-385)
# v25 pos[386:514] = v24_pos[0:128]   (新位置 386-513)
v25_pos = torch.cat([v24_pos] + [v24_pos[:128]] * 3, dim=0)
```

**训练配置**:
- B=4, T=512 (vs v24 B=16, T=128, 同 2048 tokens/step)
- 4000 步, ~12 min (vs v24 11 min, 只多 10%)
- LR=1e-4, AdamW, cosine sched
- W_KL=0.1, KL anneal 1000 steps, free bits 1.0 nat
- warm-start 295/295 权重 (除 pos 单独处理)

**训练曲线** (val_ppl 噪声较大, B=4):
| step | val_ppl | 备注 |
|---:|---:|---|
| 0 | 12.84 | warm-start 起点 |
| 250 | 3.15 | 快速下降 |
| 750 | **2.22** | (训练时最佳) |
| 1000 | 3.14 | 反弹 |
| 1750 | 2.51 | |
| 2250 | 2.27 | |
| 3000 | 2.28 | |
| 3250 | **2.17** | (训练时次佳) |
| 3500 | 2.21 | |
| 3750 | 2.52 | |

**关键**: 训练时 val_ppl 在 B=4 小批下**噪声很大** (2.17 ~ 3.70), 但**端到端 PPL 在全 1016 样本上更稳定 = 2.47**.

## 3. 端到端 PPL

```
diffusion_z:  PPL 2.4742  ← 端到端 (5步扩散)
encoder_mu:   PPL 2.4605  ← 上限
random_z:     PPL 2.4795  ← 下限
PPL 比率 (diff/enc): 1.0056
PPL 范围 (enc/rand): 0.77%
```

**关键观察**:

### 3.1 PPL 大幅改善 (-25%)

v24 → v25: PPL **3.28 → 2.47** = **-25%**.

**为什么 4x 上下文带来 25% PPL 改善**:
1. **Code 长程依赖**: Swift code 常有跨函数的引用, 长上下文让 decoder 看到完整签名/类型
2. **完整语法结构**: T=128 经常切断代码块, T=512 看到完整 class/function
3. **位置信息**: 长上下文让模型更好学习相对位置关系
4. **重复模式**: Code 充满重复 pattern, 长上下文能学到这些 pattern

### 3.2 PPL 范围缩小 (1.86% → 0.77%)

**含义**:
- v24: 短 T (128), decoder 强依赖 z (z 提供的全局信息补偿短上下文)
- v25: 长 T (512), decoder 弱依赖 z (有足够 context, 不需要 z 的全局信息)

**双刃剑**:
- ✅ PPL 大幅改善
- ❌ KR3.1 主题控制更难 (decoder 不再用 z, 主题控制失效)

### 3.3 端到端 PPL 对照

| 版本 | T | Decoder | D_Z | 训练样本 | PPL | 速度 | KR1.3 |
|---|---:|---:|---:|---:|---:|---:|---:|
| v18 | 128 | 87M | 64 | 1,893 | 17.71 | — | — |
| v20a | 128 | 229M | 64 | 1,893 | 13.05 | — | — |
| v21 | 128 | 475M | 64 | 1,893 | 5.83 | 786ms | 0.295x |
| v22a | 128 | 475M | 256 | 1,893 | 4.39 | 847ms | 0.32x |
| v23 | 128 | 475M | 256 | 6,317 | 3.37 | 761ms | 0.286x |
| v24 | 128 | 475M | 256 | 19,307 | 3.28 | 751ms | 0.282x |
| **v25** | **512** | **475M** | **256** | **19,307** | **2.47** | **828ms** | **0.311x** |
| 500M AR | 512 | 475M | — | 1,893 | 8.86 | 2665ms | 1.0x |

**v25 PPL 2.47 vs 500M AR PPL 8.86 = -72%**, 速度 KR1.3 = **0.311x** (远超 1.30x).

## 4. 速度 KR1.3

| 阶段 | 端到端 (5步+100AR) | KR1.3 vs 500M AR |
|---|---:|---:|
| 500M AR baseline | 2,665ms | 1.0x |
| v21 (T=128) | 786ms | 0.295x |
| v22a (T=128) | 847ms | 0.32x |
| v23 (T=128) | 761ms | 0.286x |
| v24 (T=128) | 751ms | 0.282x |
| **v25 (T=512)** | **828ms** | **0.311x** |

**v25 速度 828ms, 比 v24 慢 10% (77ms)**.

**为什么 T=4x 只慢 10%**:
- AR 100 步每次输入是 cur (1→100) token, **不超过 100**
- T 增大只影响 pos embedding 大小 (130→514), 不直接影响 attention
- 主要慢在 pos embedding lookup (514 vs 130 = 4x 大, 但常数项, < 1ms)
- 真实慢源: 5 步扩散略多 (~10ms), AR 略多 (~67ms, 因为 Q@K 略大)

## 5. 关键工程教训

### 5.1 上下文长度是最高边际收益的维度

```
v22a → v23 (数据 3.3x):    PPL -23%,   -0.10% PPL per 1% data
v23  → v24 (数据 3.1x):    PPL -2.7%,  -0.01% PPL per 1% data
v24  → v25 (T 4x):         PPL -25%,   显著改善
```

**含义**: 在数据饱和时, 上下文长度比数据量更值得投资.

### 5.2 PPL 范围缩小的双刃剑

- 短 T: decoder 强依赖 z, PPL 范围大, 主题控制有可能
- 长 T: decoder 弱依赖 z, PPL 范围小, PPL 改善, 主题控制难

**v25 牺牲了 KR3.1 换取 PPL 改善**. 如要恢复 KR3.1, 需要:
- 显式 cross-attention (z 作为 key/value, decoder query)
- 或 z-conditioned generation (强制 decoder 用 z)

### 5.3 评估速度不受 T 影响的发现

**关键洞察**: 评估时 AR 用 cur (1→100) token, 不超 T. 所以 T 增大不直接拖慢 AR.

这意味着:
- 训练时 T 大拖慢 (attention 是 O(T²))
- 评估时 T 大不拖慢 (cur << T)

**v25 训练 12 min 与 v24 11 min 几乎相同** (B 减半抵消了 T 增大).

### 5.4 Pos Embedding 复制策略有效

策略 A (复制 + 4x 重复) 让 v25 在 step 0 就有 PPL 12.84 (vs 随机初始化应该是 2261 chars 的随机熵, 约 PPL=2261), step 250 达到 3.15 (warm-start 有效).

## 6. v26 决策树

### 6.1 进一步扩 T 收益递减

| T | 上下文 | PPL 估 | 速度估 | KR1.3 |
|---:|---:|---:|---:|---:|
| 512 (v25) | 4x | 2.47 | 828ms | 0.311x |
| 1024 | 8x | 2.3 估 | 850ms 估 | 0.32x 估 |
| 2048 | 16x | 2.2 估 | 900ms 估 | 0.34x 估 |

**T 进一步扩**: 边际收益小 (PPL 已很低), 速度不增, 不推荐.

### 6.2 恢复 KR3.1 主题控制

- **加 cross-attention**: 在 decoder 各层加 z-conditioned attention
  - 风险: 改动大, 需重训 decoder
  - 收益: 主题控制 + 维持 PPL
- **强制 z 使用**: loss 中加 z-must-use 项
  - 简单: z 提供的全局信息应体现在 loss 中
  - 但效果可能有限

### 6.3 投机解码 (恢复速度到 v24 水平)

**核心思想**: 用小模型 draft N tokens, 大模型 1 次 forward 验证 N tokens.

**BAD-DP 适配**:
- 5 步扩散 → z0 (50ms, 不变)
- 小 decoder (100M) AR 生成 5 个 draft tokens (~50ms)
- 大 decoder (500M) 1 次 forward 验证 5 tokens (~80ms)
- 接受匹配的前缀, 否则重采样
- **总: ~180ms**, 远低于 v25 的 828ms

**预期加速**: 828ms → 300-500ms (取决于接受率 70-90%)
**训练成本**: 小 decoder 100M 训练 ~10 min, 主 decoder 不变
**风险**: 低 (主 decoder 不变, 小 decoder 失败也不影响大 decoder)

**推荐**: **v26 = 投机解码**, 把 v25 的 828ms 拉回 400-500ms.

### 6.4 平衡方案

| 路线 | 估计 PPL | 估计速度 | 估计 KR1.3 |
|---|---:|---:|---:|
| 当前 v25 | 2.47 | 828ms | 0.311x |
| v25 + 投机 (v26) | 2.47 | 400ms | 0.15x |
| v25 + 扩 T 1024 | 2.30 | 850ms | 0.32x |
| v25 + 加 cross-attn | 2.40 | 900ms | 0.34x |
| 全部 (T 1024 + 投机 + cross) | 2.20 | 450ms | 0.17x |

## 7. v26 投机解码详细设计 (建议下一步)

### 7.1 经典投机解码 (Leviathan et al. 2023)

```
1. 小模型 (draft) 自回归生成 K 个 tokens: x1, x2, ..., xK
2. 大模型 (target) 1 次 forward 验证 K 个 tokens
3. 接受匹配的前缀, 不匹配位置重采样
期望: 接受率 80-90%, 加速 2-3x
```

### 7.2 BAD-DP 适配

```
1. 5 步扩散 → z0 (50ms, 不变)
2. 小 decoder (100M) AR 生成 K=5 draft tokens (~30ms)
3. 大 decoder (500M) 1 次 forward 验证 5 tokens (~80ms)
4. 接受匹配的前缀
5. 重复直到生成 N_AR=100 tokens
```

**总时间估计** (接受率 80%):
- 5 步扩散: 50ms
- AR 100 tokens: 100/4 = 25 步 × 110ms = 2750ms... 不对

重新算. 投机接受率 r=0.8:
- 平均接受 tokens/步 = 1 + r = 1.8
- 100 tokens 需要 ~55 步
- 每步: 扩散 0ms (共享) + draft 30ms + verify 80ms = 110ms
- 总: 5ms (扩散 1次) + 55*110ms = ~500ms

实际可能 ~400-600ms, KR1.3 ≈ 0.15-0.22x.

### 7.3 训练小 decoder

**架构**: 100M decoder (12L × 768 × 12), 与 v25 encoder 同架构
**训练数据**: v25 z (256 维) + v25 data
**训练时间**: 4000 步 × B=16 × T=512, ~5-8 min (RTX 5090)
**Loss**: KL 蒸馏 (与 v25 decoder logits 对齐) 或 独立训练

**推荐**: 先独立训练, 评估接受率. 必要时 KL 蒸馏.

### 7.4 评估

- 接受率: 在 1016 val 样本上测
- 速度: 5 步扩散 + 投机 AR
- PPL: 应与 v25 几乎相同 (大 decoder 决定)

## 9. 总结

v25 完成了**上下文扩展**:

1. **上下文 4x**: T=128 → T=512
2. **PPL 大幅改善**: 3.28 → 2.47 (-25%)
3. **速度略增**: 751ms → 828ms (+10%, 远低于 4x)
4. **PPL 范围缩小**: 1.86% → 0.77% (decoder 弱依赖 z)
5. **训练时间相当**: 11 min → 12 min

**Pareto 优势继续扩大**:
- PPL 2.47 (vs 500M AR 8.86, **-72%**)
- 速度 828ms (vs 500M AR 2665ms, **0.311x**)

**关键发现**:
- **上下文长度是最高边际收益的维度** (数据饱和后)
- **评估速度不受 T 影响** (cur 始终 << T)
- **PPL 范围与 z 使用率正相关**, 长 T 削弱 z 作用

**v26 推荐**:
- **优先**: 投机解码, 把 828ms 拉回 400-500ms
- **次选**: 加 cross-attention 恢复 KR3.1
- **长期**: T 1024 + 投机 + cross-attention 全部

**CrystaLLM 当前 SOTA (v25)**:
- PPL **2.47** (vs 500M AR 8.86, **-72%**)
- 端到端 **828ms** (vs 500M AR 2665ms, **0.311x**)
- 数据 19,307 train (复用 v24)
- 模型 475M (复用 v24)
- D_Z 256 (复用 v24)
- **T 512** (本版本扩展, 4x)
